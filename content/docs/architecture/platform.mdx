The platform is the control plane for user-visible work. It owns the web experience, accepts Elite traffic at the boundary, authenticates callbacks, and turns runtime output into durable product state.

The most important architectural point is that the critical Elite endpoints documented here are file routes, not oRPC procedures. If you are debugging the relay boundary, start from those file routes first.

## Elite-facing endpoints

| Endpoint | Route type | Role |
| --- | --- | --- |
| `/api/elite/chat` | TanStack file route | Accepts the user-side request that starts Elite chat work |
| `/api/elite/ingest` | TanStack file route | Accepts relay callbacks from the runtime ingest path |
| `/api/elite/browser-events` | TanStack file route | Accepts browser-specific events on a separate channel |

## Boundary rules that matter

- `/api/elite/chat` is a file route, so it should not be documented or reasoned about as an oRPC boundary.
- `/api/elite/ingest` is also a file route and authenticates bearer callbacks through `resolveAgentCallbackAuthContext`.
- Browser activity is intentionally split out to `/api/elite/browser-events` instead of being folded into the main ingest callback path.

## What the platform does after ingest

The runtime does not send UI-ready state directly to the browser. The platform receives ingest events, authenticates them, and transforms them into durable stream events through `applyEliteRunIngestEvents` in `runs.ts`. That transformation boundary is what makes replay, recovery, and multi-surface product state possible.
