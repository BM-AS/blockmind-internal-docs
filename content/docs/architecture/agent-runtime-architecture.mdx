## Context

BlockMind serves two runtime profiles with fundamentally different longevity and compute requirements:

- **Standard tier**: bounded portfolio analyst — stateless, short-lived, scales to zero on Vercel
- **Elite tier**: persistent agent workspace — long-lived, disk-backed, stays warm on Fly Machines

We need a single runtime contract (`@blockmind/agent-runtime`) that both tiers implement, so product logic (tools, prompts, credit metering) is portable and the platform can route users to the right tier without code duplication.

This ADR replaces the implicit assumption that all agent work happens inside OpenClaw sessions.

---

## Decision

### Hard Constraints (Oracle 2026-04-02)

1. **Standard = HTTP/SSE + DB-backed runtime on Vercel**. No persistent processes, no OpenClaw session model.
2. **Elite = persistent, volume-backed OpenClaw runtime on Fly**. Remains unchanged.
3. **`@blockmind/agent-runtime` is the shared contract**. Both tiers implement the same TypeScript interface.
4. **Canonical session store = SQL event log** (PlanetScale). Replaces disk-based Pi/OpenClaw session assumptions for anything that needs replayability or cross-tier visibility.
5. **AI SDK handles model transport**. Inngest handles durable work and chat delivery.

---

## Tier Boundary

| Concern | Standard | Elite |
|---|---|---|
| Runtime host | Vercel Functions | Fly Machines |
| Session model | Stateless HTTP + SQL event log | Persistent OpenClaw + disk |
| Cold start | ~100ms (Vercel) | ~2s (Fly Machine keep-alive) |
| Max session length | 30 min (Vercel timeout) | Hours/days |
| Tool access | Subset (read-only DeFi APIs) | Full (browser, file system, code execution) |
| Scaling | Auto-scales to zero | 1:1 user:machine |
| Cost model | per-invocation (Vercel) | per-seat (Fly) |

### What defines the line

Standard is a **bounded analyst**: it receives a user message, reads current portfolio state from the DB, calls 1-3 read-only DeFi APIs, streams a response via SSE, and returns. No side effects beyond logging.

Elite is an **agent with agency**: it maintains persistent browser context, can write files, execute code, spawn Inngest workflows, and take actions across sessions.

## What Runs Where

### Browser (both tiers, client-side)

```
apps/platform/src/lib/agent-client/         ← shared client SDK
├── AgentClient          ← tier-agnostic, connects to /api/agent/chat
├── useAgent()           ← TanStack Query hook
└── useAgentStream()     ← SSE subscription hook
```

The client is identical for both tiers. It sends messages to `/api/agent/chat` and receives SSE streams. The server-side routing (Vercel vs Fly) is opaque to the client.

### Vercel Functions (Standard tier)

```
apps/platform/src/routes/api/agent/chat.ts   ← entry point, route guard
functions/agent-standard/                   ← Vercel Edge/Server Function
├── handler.ts            ← HTTP handler, parses request, routes to RuntimeHost
├── RuntimeHost.ts        ← implements @blockmind/agent-runtime
├── SessionStore.ts       ← SQL event log (PlanetScale)
├── ToolRegistry.ts       ← Standard tool set (read-only DeFi)
└── ModelTransport.ts     ← AI SDK model calls
```

**Request flow (Standard)**:
1. `POST /api/agent/chat` → Vercel Function
2. `RuntimeHost` loads conversation history from SQL event log
3. Tool execution via `ToolRegistry` (in-process, no sandbox needed)
4. AI SDK streams response via SSE
5. Each turn written to SQL event log before response
6. Function exits; Vercel scales to zero

### Fly Machines (Elite tier, unchanged)

```
apps/platform/src/routes/api/elite/chat.ts   ← entry point
functions/agent-elite/                       ← (remains as today)
├── relay.ts               ← OpenClaw relay launcher
├── elite-chat-relay-script.ts  ← Pi/OpenClaw session manager
└── (OpenClaw handles the rest)
```

**Request flow (Elite)** — unchanged:
1. `POST /api/elite/chat` → Nitro route → OpenClaw relay
2. Pi session loads workspace from disk
3. Tools execute in OpenClaw sandbox
4. OpenClaw streams via SSE to relay → client

## Runtime Contract

```typescript
// packages/agent-runtime/src/index.ts

export interface Tool {
  name: string;
  description: string;
  inputSchema: z.ZodType;
  execute: (input: unknown, context: ToolContext) => Promise<ToolResult>;
}

export interface ToolContext {
  userId: string;
  workspaceId: string;
  tier: 'standard' | 'elite';
  credits: CreditMeter;
  logger: Logger;
}

export interface CreditMeter {
  reserve(action: string, amount: number): Promise<string>; // reservationId
  settle(reservationId: string): Promise<void>;
  rollback(reservationId: string): Promise<void>;
}

export interface AgentMessage {
  id: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  createdAt: Date;
}

export interface RuntimeSession {
  id: string;
  tier: 'standard' | 'elite';
  userId: string;
  workspaceId: string;

  // Core methods both tiers implement
  send(message: AgentMessage): Promise<AsyncIterable<string>>; // streaming text
  executeTool(name: string, input: unknown): Promise<ToolResult>;
  getHistory(): Promise<AgentMessage[]>;

  // Elite-only (Standard no-ops or throws)
  getBrowserSession?(): Promise<BrowserSession>;
  writeFile?(path: string, contents: string): Promise<void>;
  readFile?(path: string): Promise<string>;
}

export interface RuntimeHost {
  tier: 'standard' | 'elite';
  createSession(userId: string, workspaceId: string): Promise<RuntimeSession>;
  listTools(): Tool[];
  destroySession(sessionId: string): Promise<void>;
}
```

### Standard tool set (MVP)

| Tool | Description | Tier |
|---|---|---|

## Session & Memory Strategy

### Standard: SQL Event Log

```sql
CREATE TABLE agent_sessions (
  id          VARCHAR(32) PRIMARY KEY,
  user_id     VARCHAR(32) NOT NULL,
  workspace_id VARCHAR(32) NOT NULL,
  tier        ENUM('standard') NOT NULL DEFAULT 'standard',
  created_at  TIMESTAMP DEFAULT NOW(),
  last_active TIMESTAMP DEFAULT NOW(),
  metadata    JSON
);

CREATE TABLE agent_events (
  id           BIGINT AUTO_INCREMENT PRIMARY KEY,
  session_id  VARCHAR(32) NOT NULL,
  message_id  VARCHAR(32) NOT NULL,
  role         ENUM('user','assistant','system','tool') NOT NULL,
  content      TEXT NOT NULL,
  tool_name    VARCHAR(64),
  tool_input   JSON,
  tool_result  JSON,
  credits_used INT,
  created_at   TIMESTAMP DEFAULT NOW(),
  INDEX (session_id, created_at),
  FOREIGN KEY (session_id) REFERENCES agent_sessions(id)
);
```

- Each conversation turn = one `agent_events` row
- `role='tool'` rows store tool calls and results
- Standard sessions are replayable from the event log
- Sessions auto-expire after 30 days of inactivity

### Elite: OpenClaw on disk (unchanged)

Elite sessions remain on the Fly Machine filesystem. The OpenClaw session store is the source of truth. A background job syncs conversation summaries to SQL for audit/billing, but the runtime itself is disk-backed.

## Transport

### AI SDK Integration (TanStack Start, not Next.js)

```typescript
// Standard tier — AI SDK in Vercel Function
import { anthropic } from '@ai-sdk/anthropic';
import { streamText } from 'ai';

const model = anthropic('claude-sonnet-4-6');

export const runtimeHost: RuntimeHost = {
  async send(message) {
    const result = streamText({
      model,
      system: SYSTEM_PROMPT,
      messages: await loadHistory(this.session.id),
      tools: toolDefinitions,
    });
    return result.fullStream;
  }
};
```

### SSE Transport (Standard)

TanStack Start supports `Response` with `ReadableStream`. The Vercel Function returns:

```typescript
return new Response(
  new ReadableStream({
    async start(controller) {
      for await (const chunk of result.fullStream) {
        controller.enqueue(`data: ${JSON.stringify(chunk)}\n\n`);
      }
      controller.close();
    }
  }),
  { headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache' } }
);
```

### Inngest for Durable Work

Long-running tool calls (portfolio analysis, news fetching) that exceed Vercel's 10s timeout are offloaded to Inngest:

```typescript
// Inngest step for heavy DeFi aggregation
inngest.createFunction(
  { id: 'portfolio-analysis' },
  { event: 'agent/analysis-requested' },
  async ({ event, step }) => {
    const positions = await step.run('fetch-positions', () =>
      portfolioService.getPositions(event.data.workspaceId)
    );
    const prices = await step.run('fetch-prices', () =>
      Promise.all(positions.map(p => priceService.get(p.token)))
    );
    return { positions, prices };
  }
);
```

## Failure Modes & Rollback

| Risk | Likelihood | Mitigation |
|---|---|---|
| Vercel cold starts > 10s | Medium | Keep 1 warm instance via cron; edge fallback |
| AI SDK token budget overruns | Medium | Per-request credit reservation; hard cap |
| SQL event log write failures | Low | Write-ahead log; Inngest retry queue |
| Standard tools insufficient for users | High | Elite escape hatch; tool portability is one-file |
| Inngest step timeouts (>10s) | Medium | Chunk work; streaming partial results |

**Rollback**: Flip tier flag in DB → user back on Elite/OpenClaw. No data migration needed since SQL log is append-only.

- [ ] **Browser events on Standard**: Can we emit browser/artifact events from Vercel, or is that Elite-only? (Oracle mentioned this as an open question.)
- [ ] **Session migration**: When a user upgrades Standard → Elite, does their conversation history migrate? How?
- [ ] **OpenAI compatibility**: Does AI SDK `anthropic` provider cover all our needs, or do we need the full `generateText` + `streamText` combo for tools?
- [ ] **Inngest sandbox**: Are Inngest steps safe for untrusted tool code, or do we need a separate execution sandbox?

## Why This Approach

The alternative — keeping a single OpenClaw-based runtime for both tiers — fails on cost and scale. OpenClaw/Pi sessions are designed for long-lived, stateful agents. For a portfolio analyst that needs 5 minutes of compute per request, spinning up a persistent Fly Machine is massive overprovision. Vercel Functions handle this naturally with per-request billing and auto-scale-to-zero.

The shared `@blockmind/agent-runtime` contract means we write tool logic once and run it on both tiers. The tier difference is purely in the `RuntimeHost` implementation — the tools, prompts, and credit metering are shared.
