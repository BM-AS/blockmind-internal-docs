Chat flow in BlockMind is a platform-coordinated handoff into the Elite runtime, followed by a structured callback stream back into the platform. The crucial detail is that the relay boundary is the ingest file route, not an oRPC endpoint.

That correction matters because the platform is doing more than proxying text. It authenticates agent callbacks, separates browser activity onto its own endpoint, and transforms raw ingest into the durable events the product actually consumes.

## Boundary summary

| Surface | Reality |
| --- | --- |
| Chat entrypoint | `/api/elite/chat` is a TanStack file route |
| Relay ingest target | Relay launches with `ingestPath: '/api/elite/ingest'` |
| Ingest auth | `/api/elite/ingest` accepts bearer auth through `resolveAgentCallbackAuthContext` |
| Browser activity | Sent to separate `/api/elite/browser-events` endpoint |
| Artifact delivery | Sent to a separate artifact endpoint |

## What the relay sends

`/api/elite/ingest` accepts a discriminated union of ingest events:

- `upstream`
- `child-session-link`
- `run-started`
- `run-usage-summary`
- `assistant-chart-candidate`
- `assistant-complete`
- `error`
- `heartbeat`
- `tool-result-pending`
- `tool-result-ready`
- `tool-result-failed`
- `research-activity`

## What the platform does next

The platform transforms ingest events into durable stream events through `applyEliteRunIngestEvents` in `runs.ts`. That is the boundary between runtime callbacks and the product state the UI can replay, recover, and render.

## Why the browser-events split exists

Browser activity is not mixed into the main ingest path. It has a separate `/api/elite/browser-events` endpoint so browser telemetry can evolve independently from the core run ingest contract.
