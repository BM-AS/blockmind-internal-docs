# Documentation Site Review

## Executive Summary
The Fumadocs/Next.js setup is fundamentally sound: the source loader, MDX wiring, docs layout, and search route are all clean and minimal, and `npx tsc --noEmit` passed during this review. The main problem is not the site framework, it is documentation freshness: several of the most important architecture pages and diagrams now describe an older system shape rather than the current production code in `~/dev/blockmind`. The biggest accuracy gaps are in the chat relay/ingest flow, the credit system, and the provisioning workflow; those three pages currently misstate core boundaries, business rules, and lifecycle steps. The information architecture is readable for internal browsing, but it does not yet separate executive-level explanations from developer-operational detail, and key topics that a new engineer would need are still missing.

## Strengths
- The Fumadocs setup is correct and appropriately simple: `next.config.mjs:1-10`, `lib/source.ts:1-7`, `app/docs/layout.tsx:1-11`, `app/docs/[[...slug]]/page.tsx:17-52`, and `components/mdx.tsx:1-17` are coherent and match the intended Fumadocs pattern.
- Navigation is shallow and easy to scan. `content/docs/meta.json:1-14` and `content/docs/architecture/meta.json:1-4` keep the main architecture material accessible without excessive depth.
- The architecture section already has the right instinct: system landscape, container views, and flow-specific pages are the correct categories for this audience.
- The docs consistently use diagrams plus prose, which is the right format for architectural onboarding.
- The platform page correctly identifies several real implementation choices, including TanStack Start, Better Auth, Inngest, and durable streams as architectural pillars: `content/docs/architecture/platform.mdx:8-29`.

## P0 Critical (blocks usefulness)
- **The chat flow page and diagrams describe the wrong relay boundary.** Docs: `content/docs/architecture/chat-flow.mdx:16-28`, `content/docs/architecture/elite.mdx:40-49`, `diagrams/d2/ChatFlow.d2:49-58`, `diagrams/d2/EliteContainers.d2:49-55`. Docs say the relay transforms OpenClaw SSE into typed durable-stream events and POSTs them to `/api/agent/events`, then the ingest endpoint appends those events directly to the durable stream. Code: `~/dev/blockmind/apps/platform/src/features/elite/chat/server/relay.ts:56-66`, `~/dev/blockmind/apps/platform/src/routes/api/elite/ingest.ts:13-42`, `~/dev/blockmind/apps/platform/src/features/elite/chat/server/ingest-events.ts:35-124`, `~/dev/blockmind/apps/platform/src/features/elite/chat/server/runs.ts:1769-1995`. Code shows the relay is launched with `ingestPath: '/api/elite/ingest'`, a separate `browserEventsUrl: '/api/elite/browser-events'`, and an artifact endpoint; the ingest route accepts intermediate ingest events such as `upstream`, `run-started`, `heartbeat`, and `assistant-chart-candidate`; the platform then transforms those ingest events into durable stream events inside `applyEliteRunIngestEvents`. This is a core architectural boundary, so the current docs are actively misleading.
- **The credit system page and diagram describe the wrong billing model.** Docs: `content/docs/architecture/credit-flow.mdx:8-37`, `diagrams/d2/CreditFlow.d2:12-60`, `content/docs/architecture/platform.mdx:20-21,39-40`. Docs say pricing is by model tier (`Flash`/`Sonnet`/`Opus`), balances are per-org, and credits are deducted after completion without up-front reservation. Code: `~/dev/blockmind/apps/platform/src/lib/credits/tiers.ts:9-30,114-156`, `~/dev/blockmind/apps/platform/src/lib/credits/classify.ts:101-193`, `~/dev/blockmind/apps/platform/src/routes/api/elite/chat.ts:378-437`, `~/dev/blockmind/apps/platform/src/lib/credits/service.ts:445-626`, `~/dev/blockmind/apps/platform/src/db/schema/billing/credit-balances.ts:13-38`, `~/dev/blockmind/apps/platform/src/lib/inngest/functions/elite-chat-settlement.ts:417-487`. Code shows action-type pricing (`chat`, `analysis`, `deep_dive`, etc.), user-scoped balances keyed by `userId`, pre-flight reservation via `reserveCredits`, and later settlement/refund via `settleReservation`. This page currently teaches the wrong business logic.
- **The provisioning flow page and diagram are stale relative to the production orchestration.** Docs: `content/docs/architecture/provisioning-flow.mdx:8-23,34-62`, `diagrams/d2/ProvisioningFlow.d2:24-113`. Docs say provisioning is an 8-step flow with identity sync as a first-class provisioning step, wake schedules for "morning brief + daily pipeline", and finalize restoring autostop. Code: `~/dev/blockmind/apps/platform/src/lib/inngest/functions/workspace-provisioning.ts:438-780`, `~/dev/blockmind/apps/platform/src/workflows/workspace-provisioning/steps.ts:1043-1181`, `~/dev/blockmind/apps/platform/src/workflows/workspace-provisioning/steps.ts:905-979`. Code shows a more complex production flow: warm-pool claim or cold create, AgentMail pod provisioning, activation, wait-for-machine-started, sync-workspace-machine-binding, parallel file writing plus gateway wait, security-config seeding, wake-schedule creation, best-effort AgentMail inbox provisioning, finalize, bootstrap event emission, claimed-pool cleanup, then autostop restore. It also explicitly comments that identity sync is not needed during provisioning because runtime files are already seeded during file writing. This is too far off to trust for onboarding or product discussion.

Docs say the chat flow goes through an oRPC "Elite Router" and relay ingest lands in an oRPC "Agent Events Router". Code: `~/dev/blockmind/apps/platform/src/routes/api/elite/chat.ts:1-41`, `~/dev/blockmind/apps/platform/src/routes/api/elite/ingest.ts:1-13`, `~/dev/blockmind/apps/platform/src/features/elite/events/router.ts:1-18,151-176`, `~/dev/blockmind/apps/platform/src/lib/orpc/router.ts:31-63`, `~/dev/blockmind/apps/platform/src/routes/api/rpc.$.ts:14-21,105-145`. Code shows chat send and ingest are TanStack file routes under `/api/elite/*`, while `eventsRouter` is a workspace event CRUD router under `/api/rpc` and is not the relay ingest surface. The current diagrams introduce conceptual container names that look authoritative but map poorly to the real codebase.
- **The authentication page is inaccurate and too generic to be useful.** Docs: `content/docs/systems/auth.mdx:8-19`. Docs say email/password auth plus Google and GitHub OAuth, secure cookies, and RBAC. Code: `~/dev/blockmind/apps/platform/src/lib/auth/index.ts:25-38,64-77,77-138,174-182,191-214`. Code shows email/password is disabled, only Google social auth is configured, and the actual auth surface includes anonymous accounts, bearer tokens, email OTP, organization support, session cookie caching, and custom user fields. The current page is not a simplification of reality; it is factually wrong.
- **Diagram freshness is not enforced by the build pipeline.** Code quality issue in docs repo: `scripts/build.sh:7-21` skips both Structurizr validation and diagram export on CI/Vercel, relying on pre-committed SVGs; `scripts/validate.sh:1-6` exists, but it is not part of the CI path. That means the site can deploy with stale or invalid diagrams even when the source DSL or D2 changed. For a docs site where diagrams carry architectural meaning, this is too weak.

## P2 Medium (improve over time)
- **Durable streams are central to the system, but they do not have a dedicated page and the existing explanation is too shallow.** Docs: `content/docs/architecture/meta.json:1-4`, `content/docs/architecture/platform.mdx:42-53`, `content/docs/architecture/chat-flow.mdx:59-61`. Code: `~/dev/blockmind/apps/platform/src/features/elite/durable-streams/chat-events.ts:9-18,171-245`, `~/dev/blockmind/apps/platform/src/features/elite/durable-streams/use-elite-stream-transport.ts:14-20,247-287,333-441,520-558`, `~/dev/blockmind/apps/platform/src/features/elite/collections/schema.ts:233-257`, `~/dev/blockmind/apps/platform/src/features/elite/collections/collections.ts:53-79`. Code shows replay-from-start, rejected-offset recovery, stream-control resume hints, idle watchdog recovery, stale-pending replay fallback, and typed projection collections. The docs mention durable streams as a concept, but not the runtime behavior that matters during failures or reconnects.
- **The docs do not explain several important production behaviors around chat runs.** Code: `~/dev/blockmind/apps/platform/src/routes/api/elite/chat.ts:378-460`, `~/dev/blockmind/apps/platform/src/features/elite/chat/server/runs.ts:1794-1844,1902-1995`, `~/dev/blockmind/apps/platform/src/lib/inngest/functions/elite-chat-settlement.ts:417-506`, `~/dev/blockmind/apps/platform/src/routes/api/elite/browser-events.ts:16-88`, `~/dev/blockmind/apps/platform/src/features/workspace/agentmail.ts:236-320`. Missing topics include child-session linking/subagent delivery, browser-event ingestion as a separate endpoint, artifact upload as a separate endpoint, reservation-vs-settlement lifecycle, and best-effort AgentMail inbox provisioning. These are architecturally important enough that their absence weakens onboarding.
- **The information architecture is readable, but the top-level sections are uneven in maturity.** `content/docs/meta.json:1-14` gives equal weight to `Architecture`, `Systems`, `Decisions`, and `Agents`, but `content/docs/systems/meta.json:1-4` contains only `auth`, and `content/docs/agents/meta.json:1-4` contains only `odin`. `content/docs/agents/odin.mdx:8-26` reads more like internal lore than system documentation. The result is a nav tree that looks comprehensive while only one section is actually developed.
- **Audience fit is mixed.** `content/docs/architecture/index.mdx:8-37` works reasonably well for a product owner, but most drill-down pages quickly assume runtime internals and implementation vocabulary. A new developer still lacks pages for session model, state transitions, failure modes, and operational boundaries; a CEO/non-technical reader lacks dedicated high-level outcome-oriented pages that explain why these architectural choices exist.
- **Container names and labels over-index on conceptual neatness rather than real ownership boundaries.** Examples: `structurizr/workspace.dsl:35,83-87,158-185` and `diagrams/d2/PlatformContainers.d2:24-60` present components such as "Agent Events Router", "Elite Router", and "Durable Streams Client" as if they are single owned modules. In code, the behavior is spread across file routes, server helpers, durable-stream runtime utilities, and collection projection modules. The diagrams should still abstract, but the abstraction should line up with code ownership.

## P3 Low (nits)
- `components/d2-diagram.tsx:15-20` uses a raw `<img>` with no explicit dimensions. This is acceptable for an internal docs site, but it gives up `next/image` optimizations and can cause layout shift on slower loads.
- `content/docs/index.mdx:8-13` is functional, but the landing page is thin relative to the importance of the docs site. A short audience-oriented overview would help frame the rest of the material.
- `content/docs/architecture/chat-flow.mdx:59-61` mentions a known SSE truncation issue and PR number, but the corresponding durable-stream recovery behavior is not documented in the durable-stream section because that section does not yet exist.

## Missing Content (prioritized)
- **1. Durable streams deep dive.** Document ingest-event types vs durable-stream event types, the transformation boundary, stream-control resume hints, replay semantics, stored offsets, stale replay fallback, and typed collection projections.
- **2. Chat run lifecycle and settlement.** Document `run.accepted -> running -> settling -> completed/refunded`, pre-flight reservation, final settlement, admin bypass, tool usage capture, and recovery when authoritative usage must be reconstructed.
- **3. Provisioning reality.** Document warm-pool vs cold-create, machine-start wait, machine-binding sync, security-config seeding, AgentMail pod vs inbox provisioning, bootstrap event emission, and why identity sync is not the primary provisioning mechanism anymore.
- **4. Auth and access model.** Document anonymous sessions, email OTP, Google OAuth, bearer tokens for callbacks, organization rules, and session cookie caching. The current page does not describe the real authentication surface.
- **5. Session and routing model.** Explain UI session keys vs gateway session keys, child-session links, relay launch payload, and the split between `/api/elite/chat`, `/api/elite/ingest`, `/api/elite/browser-events`, and `/api/agent/artifacts`.
- **6. Runtime revisions and template sync.** Document how managed workspace files, runtime revisions, and config reconciliation interact, especially because those mechanisms are central to safe rollout behavior.

## Strategic Recommendations
1. Rewrite the three stale architecture pages first: `chat-flow`, `credit-flow`, and `provisioning-flow`. Until those are fixed, the site is not a trustworthy system reference.
2. Split the docs into two explicit tracks: `Overview` pages for CEO/product discussions and `Engineering` pages for code-near architecture. Right now both audiences are partially served and neither is fully served.
3. Rename diagrams and pages to match real code boundaries. Prefer route and module names that exist in the repo over conceptual labels like "Elite Router" or "Agent Events Router" unless those abstractions are clearly marked as conceptual.
4. Add freshness enforcement to the docs pipeline. At minimum, CI should validate Structurizr DSL and fail if committed SVGs are out of date relative to their sources.
5. Treat the codebase as the canonical source of truth and make docs updates part of architecture-changing work. This site is close to being valuable, but only if it is aggressively kept synchronized with `~/dev/blockmind`.
